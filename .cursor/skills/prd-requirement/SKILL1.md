# 产品需求文档 (PRD)

## 概述

设计全面、生产级别的产品需求文档（PRD），以弥合业务愿景与技术执行之间的差距。此技能适用于现代软件系统，确保需求得到清晰定义。

## 适用场景

在以下情况下使用此技能：

- 启动新产品或功能开发周期
- 将模糊的想法转化为具体的技术规格
- 为AI驱动的功能定义需求
- 利益相关者需要项目范围的统一“事实来源”
- 用户要求“编写PRD”、“记录需求”或“规划功能”

---

## 操作流程

### 第一阶段：发现（访谈）

在编写PRD之前，必须向用户提问以填补知识空白。不要假设上下文。

**询问关于:**

- **核心问题**: 我们为什么现在要构建这个？
- **成功指标**: 我们如何知道它是否有效？
- **约束条件**: 预算、技术栈或截止日期？

### 第二阶段：分析和范围界定

综合用户的输入。识别依赖关系和隐藏的复杂性。

- 绘制 **用户流程**.
- 定义 **非目标** 以保护时间表.

### 第三阶段：技术起草

使用下面的 **严格PRD模式** 生成文档。

---

## PRD质量标准

### 需求质量

使用具体、可衡量的标准。避免使用“快速”、“简单”或“直观”。

```diff
# 模糊 (不好)
- 搜索应该很快并返回相关结果。
- 用户界面必须看起来现代且易于使用。

# 具体 (好)
+ 搜索必须在10k记录数据集的200毫秒内返回结果。
+ 搜索算法在基准评估中必须达到≥85%的Precision@10。
+ 用户界面必须遵循“Vercel/Next.js”设计系统，并实现100%的Lighthouse可访问性得分。
```

## 严格PRD模式

你 **必须** 遵循以下确切结构进行输出：

### 1. 执行摘要

- **问题陈述**: 关于痛点的1-2句话。
- **拟议解决方案**: 关于修复方案的1-2句话。
- **成功标准**: 3-5个可衡量的关键绩效指标。

### 2. 用户体验与功能

- **用户角色**: 这是为谁准备的？
- **用户故事**: `作为一个[用户]，我想要[行动]以便[受益]。`
- **验收标准**: 每个故事的“完成”定义的要点列表。
- **非目标**: 我们不构建什么？

### 3. AI系统需求（如果适用）

- **工具需求**: 需要哪些工具和API？
- **评估策略**: 如何衡量输出质量和准确性。

### 4. 技术规格

- **架构概述**: 数据流和组件交互。
- **集成点**: API、数据库和身份验证。
- **安全与隐私**: 数据处理和合规性。

### 5. 风险与路线图

- **分阶段推出**: 最小可行产品 -> v1.1 -> v2.0.
- **技术风险**: 延迟、成本或依赖失败。

## 实施指南

### 做 (始终)

- **定义测试**: 对于AI系统，指定如何测试和验证输出质量。
- **迭代**: 提交草稿并就特定部分征求反馈。

### 不做 (避免)

- **跳过发现**: 在没有至少问2个澄清问题之前，永远不要写PRD。
- **虚构约束**: 如果用户没有指定技术栈，询问或将其标记为“待定”

## 示例: 智能搜索系统

### 1. 执行摘要

**问题**: 用户难以在海量存储库中找到特定的文档片段。

**解决方案**: 一个提供带来源引用的直接答案的智能搜索系统。

**成功**:

- 将搜索时间减少50%。
- 引用准确率≥95%。

### 2. 用户故事

- **故事**: 作为一个开发人员，我想要问自然语言问题，这样我就不必猜测关键词。
- **验收标准**:
  - 支持多轮澄清。
  - 返回带有“复制”按钮的代码块。

### 3. AI系统架构

- **所需工具**: `codesearch`, `grep`, `webfetch`.

### 4. 评估

- **基准**: 用50个常见的开发人员问题进行测试。
- **通过率**: 90%必须匹配预期的引用。
